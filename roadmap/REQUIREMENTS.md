# foundry — Требования (верхнеуровнево)

> Верхний слой «что и зачем» над [ROADMAP.md](ROADMAP.md) (фазы) и
> [foundry-desktop.md](foundry-desktop.md) (приложение). Детали — там; здесь —
> назначение, потребитель, цели, состав, самосовершенствование.

---

## 1. Назначение и позиция

foundry — инструмент **дисциплины и усиления** для одного senior/staff-инженера-
архитектора. Это **не автопилот**. Он автоматизирует рутину и вбивает
CRISPY-дисциплину в детерминированные механизмы, но **мышление, решения и
ответственность остаются за человеком**. Цель — качественный код (craft,
ориентир 2–3×, не 10×-slop), а не максимальная скорость любой ценой.

Человеческие гейты — не ограничение, а **суть**: инструмент усиливает суждение
архитектора, а не заменяет его. «Read the code», а не «read the plan».

---

## 2. Потребитель

- **Первичный:** solo senior/staff/архитектор на Kotlin/Spring. Держит контроль,
  читает код, утверждает на гейтах, задаёт вкус качества.
- **Вторичные (внутри системы):** LLM-агенты потребляют skills/rules; desktop
  потребляет состояние из файлов `.foundry/`.
- **Не потребитель (пока, осознанно):** команда, джуны, полностью автономный
  CI-прогон без человека, мультитенант/SaaS.

---

## 3. Цели и не-цели

**Цели**
- Провести изменение по дисциплинированному пайплайну от идеи до PR.
- Автоматизировать рутину (скаффолд, переходы, декомпозиция, черновики
  артефактов, git-механика, синк с трекером).
- Гарантировать качество через enforcement (нельзя пропустить гейт, обойти
  изоляцию, закрыть стадию без артефакта).
- **Непрерывно улучшаться:** с каждым разом меньше правок за человеком.
- Дать наблюдаемость: где change, что делает Claude, сколько стоит.

**Не-цели**
- Полная автоматизация без человека на ключевых гейтах.
- Замена harness — Claude Code (или pi) остаётся движком, foundry — методология
  поверх.
- 10×-скорость ценой качества.
- Мультитенант/командный SaaS, облако как источник правды.

---

## 4. Принципы (инварианты)

1. State в файлах, не в памяти LLM — возобновление из артефактов.
2. Own your control flow — переходы решает state machine, не промпт.
3. Enforcement, а не «magic words» — правило вбито в механизм, не в уговор.
4. Каждый агент ≤40 инструкций.
5. Человеческий гейт — точка приложения внимания, не формальность.
6. Самоулучшение — только через approved-патчи (git-обратимо).
7. Артефакт стадии = её resume-точка.

---

## 5. Функциональные требования (что система умеет)

| Способность | Зачем | Компонент |
|---|---|---|
| Управление change-lifecycle | единица работы с состоянием и историей | state machine, tracking, CLI |
| CRISPY-конвейер стадий | дисциплинированный путь идея → код | агенты-производители + проверяющие |
| Ревью артефактов (GitHub-style) | человек держит качество, loop до устраивающего | desktop review surface, typed comments |
| Калибровка skills (learning loop) | меньше правок в следующий раз | feedback-захват + calibrator |
| Аналитика | видеть лупы, стоимость, где застревает | desktop analytics |
| Интеграции | убрать двойной ввод, замкнуть контур | Jira-мост, git/GitHub, docs в PR |
| Гибкость / escape-хатчи | не терять нижний уровень | quickfix, ручной запуск, правка в редакторе |

## 6. Из каких приложений состоит (форма)

Три плоскости, координируются через файлы стора `~/.foundry/`:

1. **foundry plugin (bash-ядро)** — ставится один раз, заводит каждому проекту
   каталог в сторе. State machine, артефакты, enforcement, CLI. **Источник
   правды** — стор `~/.foundry/`, не репозиторий проекта (раскладка —
   [components.md](components.md)).
2. **foundry-desktop (control plane)** — приложение: проекты, канбан,
   live-стрим работы Claude, ревью, калибровка, аналитика. Драйвит ядро и
   Claude; ничего не дублирует. Детали — [foundry-desktop.md](foundry-desktop.md).
3. **Claude Code (движок исполнения)** — внешний; гоняет стадии (`claude -p`).
   Связь через файлы + hooks, без тесной сцепки.

Плагин работает и без desktop (CLI/TUI); desktop — богатый вид и дирижёр над
тем же ядром.

## 7. Skills, агенты и самосовершенствование

**Агенты** — по осям: стадия × (создатель / проверяющий / разведчик /
калибровщик), с изоляцией контекста. Ростер: questioner, researcher, designer,
outliner, planner, implementor, verifier + research-scout + calibrator.

**Skills** — контракты стадий (`*-contract`) + доменные (`kotlin-*`,
`spring-*`), грузятся агентами on-demand.

### Самосовершенствование — калибровка skills (наш аналог skillOpt)

Требование: если после применения skill человек делает много правок — система
это **учитывает и дорабатывает skill**, чтобы в следующий раз правок было
меньше. Механизм:

1. **Атрибуция** — каждый артефакт помнит, какие skills/agents его произвели.
2. **Сигнал** — дельта между выводом агента и утверждённой человеком версией +
   число лупов ревью + типизированные комментарии. Это «текстовый градиент».
3. **Кластеризация** — calibrator ищет повторяющуюся правку (одна и та же
   коррекция ≥N раз после данного skill).
4. **Патч** — добавить в skill правило / пример / уточнение, в рамках бюджета
   ≤40 инструкций.
5. **Регресс-гард** — патч проверяется на корпусе прошлых change'ей: стало бы
   лучше или хуже? Защита от оверфита и дрейфа.
6. **Гейт** — человек апрувит патч (meta-review); skills под git → откат
   тривиален. Роутинг: проект-специфично → аддендум в проекте; универсально →
   глобальный skill.

**Пример (SOLID).** Ты 4 раза подряд после `solid-guide` вручную выносишь
зависимость в конструктор вместо field-injection. Calibrator замечает паттерн →
предлагает в `solid-guide` правило-пример «constructor injection, не field» →
регресс-гард подтверждает, что на прошлых change'ах это сократило бы правки → ты
апрувишь → следующий implementor уже пишет так. Лупов меньше.

**Чем отличается от автоматических оптимизаторов** (DSPy / Reflexion / skillOpt):
(а) объект оптимизации — натуральные skills, не веса модели; (б) градиент —
твои правки и комментарии, а не автометрика; (в) всегда human-gated и
git-обратимо; (г) целевая метрика — число твоих лупов, она же и аналитика.

## 8. Нефункциональные требования

- **Zero-deps ядро** (bash 3.2+), кросс-платформа (Linux/macOS), покрыто CI.
- **Локально-first, приватность** — всё на хосте и в файлах, без облака как
  источника правды.
- **Обратимость** — skills и артефакты под git; любой патч калибровки откатим.
- **Вендор** — сейчас Claude Code; ядро переносимо (pi как запасной субстрат).
- **Отзывчивость** — TUI/desktop мгновенны; стадии Claude идут в фоне.
- **Тестируемость** — ядро уже под тестами и pre-commit-гейтом.

---

## 9. Открытые решения

- Стек desktop: Compose for Desktop vs Spring Boot + SPA (на хосте, без Docker).
- Serial execution vs параллелизм фоновых стадий.
- Порог авто-применения патчей калибровки (пока — всегда через approve).
- Формат feedback-дельты (unified diff vs структурированная запись).
- Метрика качества стадии (доля approve-без-правок?) — не раньше фазы 6.
