# foundry-desktop — концепт

> Формулировка, к которой мы пришли в обсуждении. Самодостаточна: её можно
> открыть в свежей сессии и начать проектирование/сборку с нуля, не имея
> предыдущего контекста. Это **не** коммитмент — это зафиксированное
> направление и набор принятых решений + открытых вопросов.

|---|---|
| `.foundry/` под git проекта — источник правды | Правда в глобальном сторе `~/.foundry/`, раскладка — [components.md](components.md#хранилище-данных--глобальное-одна-точка-правды). В репозитории проекта не лежит ничего |
| У приложения своя локальная БД (проекция) | **Базы нет.** Индексу, тредам ревью, истории дельт и настройкам нашлось файловое место. База была бы второй копией того же |
| Стек Kotlin, выбор открыт | Swift/SwiftUI, решено 2026-07-14 |

## Одна фраза

`foundry-desktop` — управляющее десктоп-приложение (control plane) над
CRISPY-пайплайном: видит все проекты, даёт богатое ревью артефактов на каждой
стадии, координирует Claude Code. Bash-плагин `foundry` при этом **не
умирает** — он становится движком/API, который приложение вызывает.

---

## Что мы решили и почему

**Развилка «отказаться от CLI ради приложения» разрешается так: не отказ, а
инверсия роли.** Сегодня `foundry` (bash) — это движок + два фронтенда (TUI для
человека, `commands/*.md` для Claude Code). В новой картине человеческий
TUI-фронтенд отходит на второй план, а его место занимает foundry-desktop —
но движок остаётся тем же bash-ядром. Приложение **драйвит** ядро, не
переписывает его.

Почему именно так:

- **Enforcement остаётся hard и переносимым.** State machine, гейты стадий,
  правила CRISPY живут в bash (M1/M8), под git, покрыты 107 тестами и CI на
  двух платформах. Переписать это в приложении = заново доказывать
  корректность на новом стеке и потерять переносимость. Приложение только
  *зовёт* `foundry`, никогда не дублирует его логику.
- **`--plain` и чистые exit-коды уже сделаны — теперь у них появляется
  потребитель.** CLI изначально проектировался как скриптуемый (машинный
  вывод, коды `0/1/2/64`). foundry-desktop — и есть тот скрипт. Всё, что уже
  построено, сохраняется и получает второго клиента.
- **Файлы `.foundry/` — уже шина интеграции.** Инвариант «state в файлах»
  (12-FACTOR §6) окупается: любой фронтенд координируется через общее файловое
  состояние. Приложению не нужна тесная RPC-связка с Claude Code — достаточно
  читать/писать `.foundry/` и смотреть за ним (fs-watch).

## Архитектура: три слоя

```
┌─────────────────────────────────────────────────────────────┐
│  foundry-desktop  (control plane, кросс-проектный)           │
│  · список проектов, настройки; своей базы нет                │
│  · доска change'ей, детали стадий, БОГАТОЕ РЕВЬЮ артефактов  │
│  · координация Claude Code                                    │
└───────────────┬───────────────────────────┬──────────────────┘
                │ шеллит foundry --plain     │ драйвит / наблюдает
                │ (exit-коды = контракт)     │
                ▼                            ▼
┌───────────────────────────┐   ┌──────────────────────────────┐
│  ~/.foundry/ + ядро (bash) │   │  Claude Code (движок стадий) │
│  — ИСТОЧНИК ПРАВДЫ на диске│   │  · claude -p  (headless run) │
│  · state machine (M1)      │   │  · hooks → сигнал в app      │
│  · артефакты стадий        │◄──┤  · пишет артефакты через     │
│  · enforcement, feedback   │   │    тот же bash-движок         │
└───────────────────────────┘   └──────────────────────────────┘
   один стор на все проекты          per-project, по требованию
```

- **Стор `~/.foundry/`** — источник правды: state machine, артефакты,
  enforcement, feedback-дельты. Один на все проекты, в репозитории проекта не
  лежит ничего. Плагин ставится один раз, `setup` заводит проекту каталог в
  сторе.
- **foundry-desktop** — один на все проекты. Своя конфигурация, свои окна.
  Здесь происходит всё управление и ревью. Базы нет — читает стор.
- **Claude Code** — движок исполнения стадий. Приложение либо гоняет стадию
  headless (`claude -p` в каталоге проекта), либо пользователь работает в
  Claude Code интерактивно, а приложение наблюдает через fs-watch/hooks.

## Контракт интеграции (несущие правила)

1. **Источник правды — файлы `.foundry/`.** Всегда. Приложение и его БД —
   проекция поверх них.
2. **Мутации состояния — только через `foundry` CLI.** Приложение никогда не
   правит `tracking.yaml` или расположение change'а напрямую; оно шеллит
   `foundry move`, `foundry approve`, `foundry set-stage` и т.п. Так state
   machine остаётся единственным местом переходов (инвариант проекта).
3. **Чтение — можно напрямую из файлов** (для скорости рендера) плюс
   `foundry ... --plain` для того, что уже разобрано в CLI.
4. **Claude Code → app** — через hooks (на переходе стадии хук бьёт в localhost
   endpoint приложения) и/или fs-watch на `.foundry/changes/*/`.
5. **app → Claude Code** — через `claude` CLI как подпроцесс в рабочем
   каталоге проекта (headless-прогон стадии) либо просто как подсказка
   пользователю «запусти стадию» при интерактивной работе.
6. **Видимость в claude-desktop** — сессии Claude Code лежат в общем сторе
   (`~/.claude/projects/...`); десктоп-приложение Claude показывает их по
   проектам, включая запущенные снаружи через CLI. Поэтому «инициировал в
   foundry-desktop → открыл сессию в claude-desktop → вмешался руками»
   работает через общий стор, БЕЗ дистанционного управления чужим UI (на него
   нет публичного контракта — не строить на нём).

**Разделение окон** (все три смотрят на одни файлы):
- claude-desktop — смотреть процесс: как агент работает, вмешаться, докрутить;
- foundry-desktop — доска, ревью артефактов (GitHub-style; отсюда дельты для
  калибровки), аналитика. Ревью НЕ переезжает в чат — чат не даёт inline-
  комментариев/diff/типизации, теряется и качество ревью, и сигнал калибровки;
- терминал/CLI — механика руками.

## Данные: всё в файлах, базы нет (правка 2026-07-15)

Раньше здесь была таблица «файлы vs БД». Она не пережила проверку: у каждой
строки правой колонки нашлось файловое место.

| Хотели в БД | Где на самом деле |
|---|---|
| кросс-проектный индекс | `~/.foundry/cache/` — сносимая зона, пересобирается обходом стора |
| треды ревью | `stages/NN/reviews/vN.md` — рядом с тем, что ревьюят |
| история дельт | `feedback/<stage>.jsonl` |
| список проектов, настройки, тема | каталог `projects/` + настройки приложения |
| кэш рендеров | не нужен — рендерим на лету |

Правило теперь одно: **правда в файлах, всё остальное пересобирается**. База
завелась бы как вторая копия того же — а две копии одного однажды разойдутся.
Заведём, если кросс-проектный вид начнёт читать тысячи файлов на открытие.

## Стек — реальные варианты для Kotlin-инженера

Отброшено: Tauri/Electron (ты хочешь Kotlin), Kotlin/Native (LLVM-таргет — про
iOS/embedded, не десктопный UI, и Spring Boot на нём не живёт).

**Вариант A — Compose for Desktop (JVM, чистый Kotlin). Рекомендация на
долгую.**
- Один язык (Kotlin), настоящее окно, кросс-платформенно (Mac/Win/Linux на
  JVM), встроенная SQLite/H2, никакого Docker.
- Драйвит `foundry` и `claude` через `ProcessBuilder` — прямой доступ к хосту.
- Минус: нужно освоить Compose, если не знаком.

**Вариант B — SPA + Kotlin Spring Boot локально. Твой компромисс, быстрый
старт.**
- Стек твоего рабочего дня — минимум трения. Бэкенд Spring Boot держит доступ
  к файлам/CLI/git, фронт — любой SPA. Кросс-платформенно (браузер).
- Минусы: это локальное веб-приложение, не «окно»; **Docker тут скорее
  мешает** — приложению нужен доступ к хостовым файлам проектов, бинарям
  `claude`/`foundry` и git; изоляция контейнера — ровно то, что придётся
  пробивать монтированием. Запускать бэкенд как обычный процесс на хосте
  (`java -jar`) проще, чем в Docker.
- Хорош как **первый шаг**: даёт рабочее приложение быстро; позже логику ядра
  (доступ к `foundry`/файлам) можно переиспользовать при переезде на Compose.

Оба варианта — JVM/Kotlin, оба шеллят один и тот же bash-`foundry`. Выбор
между ними — про «окно сейчас» (A) против «на моём стеке за выходные» (B).

## Варианты пути (от простого к амбициозному)

Во всех живых вариантах артефакты остаются файлами в `.foundry/`, а приложение
шеллит один и тот же bash-`foundry`. Поэтому переход 1 → 2 → 3 — это смена
*витрины*, не переписывание; решение обратимо.

| # | Путь | UI ревью | Стоимость старта | Потолок |
|---|---|---|---|---|
| 1 | `$EDITOR` + генерируемый `review.html` | редактор + чтение в браузере | почти ноль | скромный |
| 2 | Локальный веб: SPA + Spring Boot **на хосте** (без Docker) | богатый, браузер | средняя, твой стек | высокий |
| 3 | Compose for Desktop (Kotlin, окно, встроенная SQLite) | богатый, окно | средняя+ (учить Compose) | высокий, кросс-платформа |
| 4 | Native Swift/SwiftUI | нативный | высокая, Mac-only, не твой стек | высокий, но лок |

Самый простой путь, не загоняющий в угол — №1 (покрывает боль ревью почти
бесплатно, двери открыты). №2 — если нужен богатый UI сейчас на своём стеке.
№3 — «настоящее окно» на Kotlin как долгая форма. №4 покупает меньше всего за
свою цену вне Swift-экосистемы.

## Отклонённые развилки (необратимые, дерутся с природой задачи)

**Спеки только в БД, ничего в папках проекта — отклонено.** Claude Code —
агент над файлами и шеллом, работает внутри каталога проекта: LLM-стадии
читают/пишут артефакты как файлы, enforcement (`allowed-tools`) — по путям
файлов. Спеки в БД → на каждой стадии обязательный мост БД→файлы→БД (постоянная
проблема двух источников правды) + потеря связи с git проекта (реасон change'а
уезжает из истории рядом с кодом). `.foundry/` — это и интерфейс к агенту, и
durable-запись под git; оставить его — то, что делает приложение простым.

**Бэкенд в Docker — отклонено.** Работа приложения — *дотянуться до хоста*:
файлы произвольных проектов, бинари `claude` (+ auth в `~/.claude`),
`foundry`, `git`, fs-watch. Docker создан *изолировать* от хоста — прямой
конфликт. Пришлось бы монтировать каталоги (изоляция обнуляется), заносить
Claude-auth в контейнер, драться с UID/permission, а на macOS Docker = Linux-VM
→ медленный fs и ненадёжный inotify через bind-mount. Простой запуск даёт и
`java -jar` на хосте — без VM-границы между приложением и всем, что оно трогает.

## Продуктовое видение (что приложение делает)

Собрано из требований + идей сверху. Порядок ≈ по потоку работы.

**Проекты и файловая система**
- Видит локальную ФС, обнаруживает проекты; выбираешь проект и **ставишь в
  него плагин `.foundry`** в один клик.
- Управляет версией плагина в каждом проекте (обновление по мере эволюции).
- Единая **очередь ревью через все проекты** — «что ждёт именно меня» в одном
  инбоксе (аналог GitHub-нотификаций); для solo с несколькими проектами,
  вероятно, главный экран.

**Канбан и наблюдение за работой**
- Доска change'ей как задач (backlog / in-progress / done / declined),
  движение — через state machine.
- **Живой стрим работы Claude**: рендер потока tool-use в реальном времени —
  видно, как change разбивается на атомарные подзадачи; кнопка прервать /
  подсказать посреди стадии.
- **Таймлайн change'а**: вся траектория (стадии, лупы, комментарии, версии
  артефактов) как прокручиваемая история — ретроспектива и durable «почему».
- Нотификации «стадия ждёт твоего gate'а» — Claude работает, тебя пингует.

**Ревью артефактов (GitHub-style)**
- Богатый рендер артефакта стадии (спеки, декомпозиция задач, код — всё, что
  является артефактом); side-by-side diff на повторном ревью («что изменилось
  с прошлого раза»).
- **Типизированные комментарии**: blocking / nit / question. «Request changes»
  собирает blocking-комментарии в промпт и отдаёт Claude на следующий луп —
  твои реплики и есть задание на переделку.
- Тредовые дискуссии с агентом по конкретному месту артефакта; loop до тех
  пор, пока качество не устроит.
- Approve с тегом «хорошо, потому что…» — позитивный сигнал, не только правки.

**Аналитика**
- **Число лупов до approve на стадию** — north-star: и метрика, и цель («лупов
  всё меньше»); тренд по времени, before/after после патча скилла.
- Стоимость: токены/cost на change и на стадию, время-в-стадии, где застревают.
- Распределение причин отклонения — куда вкладывать улучшение.

**Оркестрация**
- Всё запускается из приложения, работу делает Claude (`claude -p` в каталоге
  проекта); мутации состояния — через `foundry` CLI (state machine — источник
  переходов).

## Единый цикл: ревью = сигнал = аналитика = самоулучшение

Ключевой инсайт: ревью, «система дообучается», и аналитика — **один замкнутый
цикл**, не три фичи.

```
   твоя работа на ревью                     улучшение системы
  ┌──────────────────────┐            ┌───────────────────────────┐
  │ комментарии (typed)  │  сигнал    │ калибровка предлагает      │
  │ правки артефакта     │ ─────────► │ патч skill / lint / agent  │
  │ approve / reject +   │            │ = САМ артефакт на ревью     │
  │ причина              │            └──────────────┬────────────┘
  └──────────┬───────────┘                           │ ты апрувишь
             │                                        │ (тем же UI)
             │        метрика = число лупов           ▼
             └───────────────────────────►  в след. раз лупов меньше
```

- Твои комментарии — размеченные данные («отклонил, потому что X»).
- Число лупов — одновременно KPI аналитики и цель улучшения.
- Предложенное улучшение проходит **то же GitHub-ревью** — самоулучшение
  безопасно и обратимо, едет по тем же рельсам, что и обычная работа.
- Роутинг урока при approve: «этот проект» (аддендум `.foundry/skills/<stage>`)
  или «все проекты» (глобальный скилл).
- Регресс-гард: прошлые change'и — корпус для проверки, что новый скилл не
  оверфитит и не дрейфует.

Механика захвата (снапшот на `active → review`, дельта на `review →
completed`) — из фазы 2 roadmap; здесь она получает витрину.

## Гибкость: приложение — равноправный вид, не тюрьма

Гарантия против «потерять в гибкости»: всё, что делает приложение, делается и
из CLI; правда — в файлах `.foundry/`; в любой момент проваливаешься в терминал
или свой редактор, приложение подхватывает. Escape-хатчи: любую стадию
запустить вручную / пропустить / переопределить, `/quickfix` в обход конвейера,
править артефакт в своём редакторе, подкрутить промпт стадии под проект.
Приложение добавляет удобство, но не отнимает нижний уровень.

## Последовательность — обязательный параллельный трек

**Решение: ревью обязателен → приложение закладывается в план сразу**, как
параллельный трек с вехами, синхронизированными с фазами ядра (таблица вех —
в [ROADMAP.md](ROADMAP.md), секция «Параллельный трек»).

- **D0 — спайк петли** (параллельно фазе 2): fs-watch `.foundry/changes/*/`,
  рендер одного артефакта, approve через `foundry approve …` (виден флип
  state machine), прогон одной стадии через `claude -p`. Де-риск связки —
  самое неизвестное не UI, а петля.
- **D1 — Review MVP** (готов к фазе 3, обязателен к фазе 4): change'и
  проекта, артефакт стадии, diff снапшот↔текущий, комментарии с метками,
  approve / request-changes. Ревью-поверхность существует к моменту первых
  LLM-артефактов.
- **D2 — оркестрация** (фазы 5–6): запуск стадий из приложения, live-статус,
  доска.
- **D3 — калибровка UI** (с первым calibrate, фаза 4+): очередь дельт,
  черновики патчей, approve.
- **D4 — аналитика + мультипроект** (после фазы 6): счётчик лупов, инбокс.

Правило синхронизации: веха приложения должна успевать к фазе ядра, которая
производит её материал; методология при этом не останавливается — dogfood и
фаза 2 идут параллельно D0. Ревью в `$EDITOR` остаётся escape-hatch'ем
(механика снапшот/дельта/вердикт та же), целевая поверхность — приложение.
