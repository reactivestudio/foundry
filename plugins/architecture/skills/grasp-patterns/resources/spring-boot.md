# GRASP — Spring Boot

How Spring Boot conventions implement each GRASP pattern. Most idiomatic Spring code IS GRASP-compliant by construction; the patterns describe what the framework already does. The trap is misapplying Spring features in ways that violate GRASP — fat `@Service` god classes, direct calls instead of events, vendor SDK leaking through controllers.

For pattern definitions, see `theory.md`. For Kotlin idioms, see `kotlin.md`. For violations and fixes, see `bad-practices.md`.

---

## 1. Information Expert in Spring

Spring is neutral on Info Expert — `@Entity` classes can be anaemic carriers or behaviour-rich aggregates. The discipline is on you (and on `clean-code-objects-and-data` / `ddd-tactical-patterns`).

### The Spring trap: anaemic `@Entity` + god `@Service`

```kotlin
@Entity
class Order(
    @Id val id: UUID,
    @OneToMany(...) val items: List<OrderItem>,
    var status: String,
)
// (no methods other than getters/setters generated by Kotlin properties)

@Service
class OrderService(...) {
    fun calculateTotal(orderId: UUID): Money {
        val order = orderRepo.findById(orderId).orElseThrow()
        return order.items.fold(Money.ZERO) { acc, item -> acc + Money(item.unitPriceMinor * item.quantity, ...) }
    }
}
```

The data lives on `Order` and `OrderItem`. The computation lives on the service, reaching through getters. Info Expert violated.

### The fix

Push `total()` onto `Order`. `Order` becomes behaviour-rich; the service simplifies to `order.total()`.

If `Order` is an aggregate root in DDD terms, this is straightforward — see `ddd-tactical-patterns`.

### `@Transactional` on the application service, not on the entity

The Info Expert pattern doesn't preclude using `@Transactional` at the service layer. The service orchestrates the transaction boundary; the entity owns the computation. Both can be true.

## 2. Creator in Spring

Spring's `@Component` / `@Service` are themselves applications of Creator: the framework decides who creates beans, freeing your code from `new Service(...)` calls.

For domain entities, use companion-object factories (see `kotlin.md`); Spring doesn't create domain entities, you do.

### Aggregate factories live on the aggregate, not on a `@Component`

```kotlin
class Order private constructor(...) {
    companion object {
        fun create(id: OrderId, customerId: CustomerId, items: List<OrderItem>): Order { ... }
    }
}

@Service
class PlaceOrderHandler(...) {
    fun handle(cmd: PlaceOrderCommand): OrderId {
        val order = Order.create(OrderId.new(), cmd.customerId, cmd.items)
        orders.save(order)
        return order.id
    }
}
```

The Creator of `Order` is `Order.create` (Creator GRASP). The Spring service uses it but isn't itself the Creator.

### The trap: `OrderFactory` as a Spring bean

```kotlin
@Service
class OrderFactory(...) {
    fun create(customerId: CustomerId, items: List<OrderItem>): Order { ... }
}
```

This is GRASP Pure Fabrication, not Creator. Sometimes appropriate (when creation needs DI'd dependencies — pricing, validation), but unnecessary when the data alone is enough. Default to `companion object`; promote to a Spring factory only when DI is needed at construction time.

## 3. Controller in Spring

Spring's `@RestController` IS the GRASP Controller, with the framework providing the routing, parameter binding, and response serialisation.

### Use-Case Controller per endpoint (preferred at scale)

```kotlin
@RestController
@RequestMapping("/api/v1/orders")
class OrderController(
    private val placeOrder: PlaceOrderHandler,
    private val getOrder: GetOrderHandler,
    private val cancelOrder: CancelOrderHandler,
) {
    @PostMapping
    fun place(@Valid @RequestBody req: PlaceOrderRequest): ResponseEntity<OrderResponse> {
        val orderId = placeOrder.handle(req.toCommand())
        return ResponseEntity.created(URI("/api/v1/orders/$orderId"))
            .body(OrderResponse(id = orderId))
    }

    @GetMapping("/{id}")
    fun get(@PathVariable id: UUID): OrderResponse =
        getOrder.handle(OrderId(id))?.toResponse() ?: throw NotFoundException()
}
```

The controller's responsibility: receive HTTP, validate, delegate to the use-case handler, format response. **No business logic.**

Each `*Handler` is itself a Use-Case Controller in GRASP terms — one class per use case, one reason to change.

### Facade Controller for small subsystems

For a small subsystem (a few endpoints, one team owns it), a single `OrderController` with the logic delegated to a single `OrderService` is fine. Don't pre-emptively split when the system is small.

### The trap: fat controller

```kotlin
@PostMapping
fun place(@RequestBody req: PlaceOrderRequest): OrderResponse {
    if (req.items.isEmpty()) throw BadRequest()       // validation
    val priced = req.items.map { it.copy(price = pricingClient.lookup(it.productId)) }   // pricing logic
    val order = orderRepo.save(Order(...))            // persistence
    emailClient.send(...)                              // notification
    return OrderResponse(...)
}
```

Five responsibilities in one method, in the controller layer. Pull each into the right place: validation in the DTO / Bean Validation; pricing in `PricingService`; persistence in the use-case handler; notification via event.

## 4. Low Coupling in Spring

Spring's eventing system is the canonical Low Coupling tool.

### `ApplicationEventPublisher` + `@ApplicationModuleListener`

```kotlin
@Service
class PlaceOrderHandler(private val events: ApplicationEventPublisher) {
    fun handle(cmd: PlaceOrderCommand): OrderId {
        val order = Order.create(...)
        orders.save(order)
        events.publishEvent(OrderPlaced(order.id, order.customerEmail, order.total()))
        return order.id
    }
}

@Component
class OrderEmailNotifier(private val email: EmailSender) {
    @ApplicationModuleListener
    fun on(event: OrderPlaced) {
        email.send(event.customerEmail, "Your order #${event.orderId}")
    }
}

@Component
class OrderAnalyticsRecorder(private val analytics: AnalyticsClient) {
    @ApplicationModuleListener
    fun on(event: OrderPlaced) {
        analytics.record("order.placed", mapOf("amount" to event.total))
    }
}
```

`PlaceOrderHandler` knows nothing of `OrderEmailNotifier` or `OrderAnalyticsRecorder`. Adding an SMS notifier is a new class; nothing else changes.

`@ApplicationModuleListener` (Spring Modulith) handles the transactional handoff (publishes after commit, retries on failure via the outbox). Use it instead of `@EventListener` for module-internal communication.

### `Map<String, T>` / `List<T>` injection for plug-in collaborators

When you need a *family* of bean implementations (one per variant), inject the collection:

```kotlin
@Service
class PaymentProcessor(
    private val gateways: Map<String, PaymentGateway>,   // key = bean name
) {
    fun process(method: String, amount: Money): Result<TransactionId> =
        (gateways[method] ?: throw UnknownPaymentMethod(method)).charge(amount)
}
```

Adding `BraintreeGateway : PaymentGateway` adds a Spring bean; `PaymentProcessor` finds it at startup. Low coupling between the orchestrator and the providers.

### When NOT to use events

Direct synchronous calls are fine — even preferable — when:
- The caller depends on the result (events are fire-and-forget).
- The reaction is intrinsic to the operation, not a side effect (e.g., debiting the wallet IS placing the order, not a reaction to it).
- The number of "listeners" is one and stable.

Don't use events as decoration; use them where genuine independence exists.

## 5. High Cohesion in Spring

Spring rewards High Cohesion — the framework happily handles 100 small `@Service` beans. The trap is the lure of the god service.

### One `@Service` per focused responsibility

```kotlin
@Service class PricingService(...) { fun price(cart: Cart): PricedCart }
@Service class TaxCalculator(...) { fun apply(amount: Money, country: Country): Money }
@Service class InvoiceRenderer(...) { fun render(order: Order): ByteArray }
@Service class OrderEmailFormatter(...) { fun format(order: Order): String }
```

Each one does one thing. Cost of having four beans vs one god bean: zero in DI, ~5× less in test setup.

### The trap: god `@Service`

```kotlin
@Service
class OrderService(
    private val orderRepo: OrderRepository,
    private val pricingClient: PricingClient,
    private val emailClient: EmailClient,
    private val notifications: NotificationService,
    private val payment: PaymentGateway,
    private val analytics: AnalyticsClient,
    private val inventory: InventoryService,
    private val tax: TaxCalculator,
) {
    fun placeOrder(...) ; fun cancelOrder(...) ; fun shipOrder(...) ; fun refundOrder(...)
    fun calculatePrice(...) ; fun calculateTax(...) ; fun checkInventory(...) ; fun ...
}
```

Eight unrelated dependencies, twelve methods spanning multiple concerns. High Cohesion violated; SRP violated; testing is a nightmare.

### The fix: split per use case + per fabrication

```kotlin
@Service class PlaceOrderHandler(...) { fun handle(cmd: PlaceOrderCommand): OrderId }
@Service class CancelOrderHandler(...) { fun handle(cmd: CancelOrderCommand) }
@Service class ShipOrderHandler(...) { fun handle(cmd: ShipOrderCommand) }
@Service class RefundOrderHandler(...) { fun handle(cmd: RefundOrderCommand) }
@Service class PricingService(...) { fun price(cart: Cart): PricedCart }
@Service class InventoryService(...) { fun reserve(items: List<Item>): Reservation }
```

Each is high-cohesion. The use-case handlers compose the focused services; the focused services don't know about each other.

## 6. Polymorphism in Spring

Combine sealed Kotlin types with Spring's bean injection for polymorphic strategies.

### Strategy pattern as `Map<Type, Bean>`

```kotlin
interface PaymentGateway {
    fun charge(amount: Money, customer: Customer): Result<TransactionId>
}

@Component class StripeGateway(...) : PaymentGateway { ... }
@Component class AdyenGateway(...) : PaymentGateway { ... }
@Component class PayPalGateway(...) : PaymentGateway { ... }

@Service
class PaymentProcessor(private val gateways: List<PaymentGateway>) {
    fun process(method: PaymentMethod, amount: Money, customer: Customer): Result<TransactionId> =
        gateways.first { it.supports(method) }.charge(amount, customer)
}
```

Adding `BraintreeGateway : PaymentGateway` adds one Spring bean. Polymorphism + DI = OCP at the framework level.

### `@Qualifier` for naming when bean type alone isn't enough

```kotlin
@Service
class PaymentProcessor(
    @Qualifier("primary") private val primary: PaymentGateway,
    @Qualifier("fallback") private val fallback: PaymentGateway,
) { ... }
```

Use sparingly — name-based DI is fragile. Prefer typed strategy interfaces.

## 7. Pure Fabrication in Spring

Most Spring `@Service` beans are Pure Fabrications — they're not domain entities, they hold focused orchestration. This is fine and idiomatic.

### Application service as Pure Fabrication

```kotlin
@Service
class OrderEmailNotifier(
    private val email: EmailSender,
    private val templates: EmailTemplateEngine,
) {
    fun notifyPlaced(event: OrderPlaced) { ... }
    fun notifyCancelled(event: OrderCancelled) { ... }
    fun notifyShipped(event: OrderShipped) { ... }
}
```

Not an entity; not infrastructure. A focused fabrication that orchestrates email-related operations triggered by domain events.

### Domain service vs application service

In DDD vocabulary:
- **Domain service** = a Pure Fabrication living in the domain layer (e.g., `PricingPolicy.applyDiscount(...)`). Stateless, no Spring dependencies, expresses domain rules.
- **Application service** = a Pure Fabrication living in the application layer (e.g., `PlaceOrderHandler.handle(...)`). Orchestrates infra; depends on Spring; uses domain services and aggregates.

Both are Pure Fabrications in GRASP terms. The DDD distinction is which layer they live in — see `ddd-tactical-patterns`.

### The trap: Pure Fabrication used as a junk drawer

`OrderUtils`, `Helpers`, `CommonComponent`. Same High Cohesion violation as in pure Kotlin — the fabrication has multiple responsibilities. Split.

## 8. Indirection in Spring

Spring DI is itself an Indirection mechanism — consumers depend on interfaces; the container provides implementations. Use that machinery deliberately.

### Domain port + infrastructure adapter

```kotlin
// domain/
interface PaymentGateway {
    fun charge(amount: Money, customer: Customer): Result<TransactionId>
}

// infrastructure/
@Component
class StripePaymentGateway(...) : PaymentGateway { ... }

// application/
@Service
class PlaceOrderHandler(private val payment: PaymentGateway) {
    fun handle(cmd: PlaceOrderCommand): OrderId {
        ...
        payment.charge(order.total(), cmd.customer)
    }
}
```

`PlaceOrderHandler` doesn't import Stripe. The interface is the indirection; Spring wires the impl at startup.

### Spring Modulith events as Indirection

Events are indirection on the *control flow* axis (not on the dependency axis):

```kotlin
@Service
class PlaceOrderHandler(private val events: ApplicationEventPublisher) {
    fun handle(cmd: PlaceOrderCommand): OrderId {
        ...
        events.publishEvent(OrderPlaced(...))
    }
}

@Component
class OrderEmailNotifier {
    @ApplicationModuleListener fun on(event: OrderPlaced) { ... }
}
```

`PlaceOrderHandler` doesn't know about `OrderEmailNotifier`. The event is the indirection; subscribers are wired by Spring at startup.

### `@RestControllerAdvice` as Indirection on error handling

```kotlin
@RestControllerAdvice
class GlobalExceptionHandler {
    @ExceptionHandler(NotFoundException::class)
    fun onNotFound(ex: NotFoundException): ProblemDetail = ...
}
```

Controllers throw domain exceptions; they don't know how those translate to HTTP. The advice is the indirection; the translation is a separable concern.

## 9. Protected Variations in Spring

Spring's profile / conditional bean / `@Bean` configuration mechanisms ARE Protected Variations machinery.

### `@Profile` swap for vendor variation

```kotlin
@Component @Profile("!test")
class StripePaymentGateway(...) : PaymentGateway { ... }

@Component @Profile("test")
class FakePaymentGateway : PaymentGateway { ... }
```

Stripe → Adyen migration: add `@Component @Profile("adyen") class AdyenPaymentGateway`, switch profile. Other code untouched.

### `@ConditionalOnProperty` for finer-grained switching

```kotlin
@Component
@ConditionalOnProperty("payment.provider", havingValue = "stripe")
class StripePaymentGateway(...) : PaymentGateway { ... }

@Component
@ConditionalOnProperty("payment.provider", havingValue = "adyen")
class AdyenPaymentGateway(...) : PaymentGateway { ... }
```

Switch via `application.yml`: `payment.provider: stripe`. PV via configuration, not deployment.

### `@Bean` in `@Configuration` to hide vendor-class construction

```kotlin
@Configuration
class GatewayConfig {
    @Bean
    fun paymentGateway(props: GatewayProps): PaymentGateway = when (props.provider) {
        "stripe" -> StripePaymentGateway(stripeApi(props))
        "adyen"  -> AdyenPaymentGateway(adyenApi(props))
        else -> throw IllegalStateException("Unknown provider: ${props.provider}")
    }
}
```

Consumers depend on `PaymentGateway`. Spring sees only the interface. Vendor-specific construction is centralised.

### API versioning as PV

```kotlin
@RestController @RequestMapping("/api/v1/orders") class OrderControllerV1(...)
@RestController @RequestMapping("/api/v2/orders") class OrderControllerV2(...)
```

Clients on `/v1` are protected from `/v2` changes. Internally the two controllers can share the use-case handlers; only the request/response DTOs differ.

### The trap: PV applied everywhere

Every interface has a cost — DI complexity, mental overhead, indirection in stack traces. Apply PV at boundaries where you have *good reason* to expect change. Don't introduce it for every internal collaboration.

## Spring features that serve GRASP at a glance

| Spring feature | Patterns served |
|---|---|
| `@RestController` / `@RequestMapping` | Controller (handles system events from HTTP) |
| Constructor injection | Indirection (depend on interfaces), Low Coupling (only inject what you call) |
| `@Service` with focused methods | Pure Fabrication (orchestration that no entity owns), High Cohesion |
| `@Component` with no methods → 100 small beans | High Cohesion (one purpose per bean) |
| `ApplicationEventPublisher` + `@ApplicationModuleListener` | Low Coupling (publisher doesn't know listeners), Indirection (the event is the seam) |
| `Map<String, T>` / `List<T>` injection | Polymorphism (orchestrator + family of impls) |
| `@Profile` / `@ConditionalOnProperty` | Protected Variations, Polymorphism |
| `@Bean` in `@Configuration` | Indirection (consumers depend on interface; factory hides vendor) |
| `@RestControllerAdvice` + `@ExceptionHandler` | Indirection (controllers don't translate exceptions), OCP |
| Spring Modulith application modules | High Cohesion at module scale, Indirection between modules via events |
| ArchUnit / Modulith fitness tests | Enforce GRASP at CI: dependency direction, cross-module coupling, layer purity |

## Spring traps that violate GRASP

| Trap | Pattern violated | Fix |
|---|---|---|
| God `@Service` with > 5 dependencies | High Cohesion, Pure Fabrication misuse | Split per use case |
| Direct `EmailClient` call in `@Service` | Low Coupling | Emit event; let listener react |
| Anaemic `@Entity` + computation in `@Service` | Information Expert | Push computation onto the entity |
| Fat `@RestController` with business logic | Controller misapplied (not just delegation) | Extract use-case handlers |
| `new VendorClient()` in business code | Indirection, Protected Variations | Wrap behind a domain port |
| `@Autowired ApplicationContext` for service-locator pulls | Indirection (broken DI) | Inject what you need at construction |
| `@Component class OrderUtils` | High Cohesion | Split into focused classes |
| Premature `@Profile` everywhere | Protected Variations over-applied | Apply only at real seams |
